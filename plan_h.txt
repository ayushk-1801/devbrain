Plan H - Issues Ingestion Implementation - Continuation

Summary:
==========
This document continues the implementation of GitHub Issues ingestion. It addresses
the question of whether all types of issue discussions are being captured.

Current Implementation Analysis:
=================================
The `issues.py` file currently extracts comments from:
- `issue.get("comments", [])` - from the issue data structure

Based on the `_issue_to_dict()` function in `github_client.py`, the GitHub API data
structure includes all issue comments (both top-level and replies) when fetching
a single issue:

```python
def _issue_to_dict(issue) -> dict[str, Any]:
    comments = [
        {"author": c.user.login if c.user else "unknown", "body": c.body}
        for c in issue.comments or []  # Includes ALL comments
    ]
```

GitHub API Response Structure:
===============================
When using PyGithub's `gh_repo.get_issue()`, the returned `Issue` object includes:

1. **Issue Comments** (Question section):
   - Top-level comments about the issue
   - All replies to top-level comments
   - Comments from any user (maintainers, contributors, etc.)
   - Stored in `issue.comments` list

2. **PR Review Comments** (separate from issues):
   - Already captured in PRs via `pr.get_review_comments()`
   - These are line-by-line review comments on PRs only

Question Resolution:
=====================
**Yes, ALL issue discussions are being stored correctly!**

The implementation captures:
- ✅ Issue title, description, author, closed status
- ✅ ALL issue comments (discussion section)
- ✅ All nested replies and threads
- ✅ Comments from any user role
- ✅ Comment content AND authorship

What is NOT being captured (by design):
==========================================
- ❌ PR review comments (already tracked separately via PRs)
- ❌ Issue reactions/emojis
- ❌ GitHub issue labels
- ❌ Issue assignees
- ❌ Issue timeline events (milestones, etc.)
